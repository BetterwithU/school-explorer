## 1. 증상 (Symptom)
- 대시보드(hq.html)에서 **📢 전체 공지**를 발송해도 **학생 기기(play.html)에 공지 배너가 안 뜬다.**
- 게임 종료·소집(announce `end:true`)도 같은 경로를 타므로 동일 위험. (종료 자체의 시트 기록은 별개로 정상 작동 확인됨)

## 2. 재현 (Repro)
1. 로그인 모드로 배정된 B1 게임을 학생 기기에서 @snu 로그인 후 플레이 중.
2. 교사가 대시보드에서 전체 공지 발송.
3. 기대: 학생 화면 상단에 `📢 …` 배너가 뜨고 10초 깜박→30초 후 사라짐.
4. 실제: **아무것도 안 뜸.**

## 3. 데이터 흐름 & 관련 코드 (경로는 정상)
**발송(교사):**
- `games/B1/hq.html` — 📢 버튼 → `window.BSync.sendAnnounce(text, end)`
- `core/sync.js:122-126` `sendAnnounce()` → `sessions/{session}/announce` 에 `{text, end, ts}` 기록

**수신(학생):**
- `games/B1/play.html:192` `initSync()` 호출 → `:262-280` 내부에서 **online일 때만** 구독 등록
- `:274-276` `window.BSync.subscribeAnnounce(a => showAnnounce(a))`
- `core/sync.js:132-134` `subscribeAnnounce()` = `onValue(sessions/{session}/announce, …)`
- `games/B1/play.html:287-328` `showAnnounce()` — 배너/오버레이 렌더 (**코드 정상 확인**: `a.text`면 배너 `display:block` + blink 타이머)

→ **발송·수신·렌더 코드 자체엔 결함 없음.** 막히는 곳은 그 사이의 **Firebase 읽기 권한**.

## 4. 진단 (Diagnosis) — 유력순
### (1순위) 배포된 Firebase 보안규칙이 학생 읽기를 막음 ★
- `database.rules.json`(레포)의 announce 규칙은 학생 읽기를 **허용**함:
  ```
  "announce": {
    ".read": "$session.matches(/^login__/) && auth != null && auth.token.email.matches(/@snu[.]ms[.]kr$/)",
    ".write": "…교사 3계정만…"
  }
  ```
  (RTDB 규칙: 부모 `sessions/$session`의 read가 교사 전용이어도, 자식 announce에서 read를 별도로 grant하면 학생이 읽을 수 있음 — 규칙상으로는 문제없음.)
- **핵심 함정:** `database.rules.json`은 레포에 있다고 **자동 배포되지 않음.** Firebase 콘솔/CLI로 **게시(publish)** 해야 실제 적용됨.
  → 레포 규칙엔 학생 read가 있는데 **콘솔에 게시된 규칙이 옛 버전(학생 read 없음)**이면, 학생의 `onValue(announce)`가 **permission_denied**로 조용히 실패 → 배너 안 뜸.
- 이건 이전에 겪은 **LIVE(teams) 미수신 = 혼잡분산 안 됨** 문제와 **같은 뿌리**(학생이 세션 실시간 노드를 못 읽음). 커밋 `9ffdb66 "보안규칙 학생 read 복원"`이 레포는 고쳤어도 **콘솔 게시가 안 됐으면 여전히 막힘.**

### (2순위) 구독 타이밍/인증
- `subscribeAnnounce`가 `bsync-ready` 시점에 등록되는데, 학생 로그인 토큰이 붙기 전이면 초기 read가 denied 될 수 있음. RTDB는 auth 변경 시 재평가·재전달하므로 보통 회복되지만, 엣지케이스로 남을 수 있음.

### (3순위) online 판정
- `play.html:264` 구독은 `isOnline()`일 때만 등록. 로그인 모드면 online=true라 정상. (open/test 모드는 애초에 공지 미지원 — 정상)

## 5. 확정 방법 (고치기 전에 반드시 이걸로 원인 못박기)
학생 기기(또는 학생 세션으로 로그인한 브라우저)에서 **F12 → Console**:
- `permission_denied at /sessions/…/announce` 류 에러 있음 → **1순위 확정**(규칙 게시 문제).
- 에러 없이 값은 도착하는데 배너만 안 뜸 → 렌더 문제로 방향 전환(단, 코드상 가능성 낮음).
- 참고: 같은 방식으로 `/sessions/…/teams` `/override` 도 denied면, 세션 실시간 노드 전반이 막힌 것 = 규칙 미게시 확정.

## 6. 해법 (Solution)
1. **Firebase 콘솔 → Realtime Database → 규칙(Rules)** 열어 현재 게시본과 레포 `database.rules.json`을 **대조.** 다르면 레포 내용으로 **게시(publish).**
   - (CLI 쓰면) `firebase deploy --only database` — 단, 프로젝트 연결·로그인 필요.
2. 게시 후 학생 기기에서 재현: 교사가 공지 발송 → **~1초 내 배너** 뜨는지 확인.
3. 여전히 안 되면 **2순위(타이밍)** 대응: `play.html`에서 `window.BAuth.onChange`로 로그인 성립 후 `subscribeAnnounce`를 (재)등록하도록 보정.
4. 회귀 방지: `announce`처럼 **학생이 읽어야 하는 노드**(`announce`, `override`, `pairs`)의 read 규칙이 콘솔에 확실히 게시돼 있는지 체크리스트화.

## 7. 검증 (Verify)
- 교사: 대시보드에서 공지 3종(5분 남음/본부 소집/잠깐 멈춰) 각각 발송.
- 학생 기기: 배너 표시 → 10초 깜박 → 30초 후 사라짐 확인.
- 종료·소집(`end:true`): 학생 화면에 🏁 종료 오버레이 뜨는지 확인.
- ⚠️ 검증은 **배포본(HTTPS)** + 등록 교사 로그인 + 실제 학생 @snu 로그인으로. 로컬(127.0.0.1)은 로그인 안 됨.

## 8. 주의
- **라이브 게임 중 보안규칙을 건드리면** 대시보드 추적·결과 저장까지 흔들릴 수 있음. 규칙 게시는 **게임이 없는 시간에**, 게시 직후 대시보드/기록이 정상인지 즉시 확인.
- 규칙 변경 시 학생 read(announce/override/pairs)와 교사 write를 **둘 다** 보존할 것.
